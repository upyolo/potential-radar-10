下面是基于两份文档提炼的完整前端提示词，可直接用于驱动前端开发（已适配 React + Vite + TypeScript + Tailwind 技术栈，去除原文档中 Next.js 假设，前后端联调通过独立 API 完成）。

前端提示词：

产品名称：璞见 看见关键词之外的真实潜力

一、产品定位与一句话目标

构建一个单页 Web 工作台，演示「从 JD 任务反推能力画像 → 从简历中寻找能力证据 → 找回被传统关键词筛选漏掉的高潜候选人」的完整闭环。核心不是"更快筛简历"，而是让招聘方"看见"传统关键词规则看不见的人，并提供可解释的证据、风险与面试追问。

目标受众：校招 HR、实习招聘负责人、新型复合岗位的用人经理、招聘评委。

二、信息架构与页面结构

整体为单页工作台（无登录、无多页路由切换），自上而下分为 5 个核心区域 + 1 个抽屉：

1. 顶部导航栏 Header

左侧：产品 Logo + 名称「潜力雷达 / Potential Radar」+ Slogan「发现被关键词筛掉的高潜候选人」

右侧：语言切换 / 状态徽章（在线模型 / 缓存演示 / 离线 Demo 模式）

2. 模块 A — JD 展示区（岗位卡片）

岗位名称、岗位类型（如：AI 产品运营实习生）

岗位描述（可折叠的完整 JD 原文）

关键词标签云（学校/专业/技能/经验关键词，作为传统筛选规则）

操作：「查看完整 JD」「编辑筛选规则」（演示用，可只读）

3. 模块 B — 流程进度条 / 操作中枢

水平 3 步骤进度指示：

加载样例数据 → 2. 运行传统关键词筛选 → 3. 启动潜力雷达 AI 分析

主操作按钮（粘性悬浮）：

第一阶段：「运行传统筛选」

第二阶段：「启动潜力雷达」（带磁吸光效，强调 AI 价值点）

重置按钮：「重新演示」

4. 模块 C — 候选人池（列表/网格）

每张候选人卡片展示：

头像（首字母色块）/ 化名编号（C001 …）

学校 / 专业 / 年级 / 期望岗位

关键词命中数（如 2/8）

状态徽章：

「通过关键词筛选」绿色

「被关键词筛掉」灰色

「AI 建议复核」金色发光高亮 ← 核心视觉焦点

卡片整体支持点击 → 打开详情抽屉

布局：响应式网格，桌面 3-4 列，移动端单列。

5. 模块 D — AI 分析结果区（结构化展示）

分为 4 个子卡片，横向或两栏布局：

D1 岗位能力画像（雷达图）

核心任务 Bullet List

5-6 维能力雷达图（如：信息整理 / 用户理解 / 数据分析 / 跨团队沟通 / AI 工具使用 / 任务拆解）

证据关键词 Chip 列表

D2 传统筛选对照面板

通过 / 漏筛 数字对比

"其中 N 位被 AI 找回" 强调数字

D3 被找回候选人排名榜

推荐等级：高 / 中 / 低（彩色徽章）

一句话推荐理由

"为什么被关键词漏掉"小标签

D4 风险与追问预览

全局风险提示卡（不能仅作录用依据声明）

6. 模块 E — 候选人详情抽屉（右侧 Sheet）

点击候选人后从右侧滑出，宽度桌面 ≈ 560px：

顶部：候选人基本信息 + 关闭

Tab 切换：

能力证据：每项能力 → 简历原文引用（高亮卡片）+ 置信度（high/medium/low）

匹配理由：为什么被关键词漏筛、为什么仍值得复核

风险提示：能力差距、证据不足点

面试追问：可直接复制的 5-8 个问题列表，每条带「复制」按钮

底部行动条：「加入面试池」「跳过」「复制完整分析」

7. 全局状态层

Loading 骨架屏：AI 分析中显示进度提示「正在拆解 JD … 抽取简历证据 …」

Toast：成功 / 模型超时 / 缓存兜底

空态：未运行分析前的引导提示

三、核心功能清单

#功能描述F1样例数据加载从静态 JSON 读取 1 个 JD + 5-8 份脱敏简历F2传统关键词筛选纯前端计算关键词命中并区分通过/漏筛人群F3一键调用 AI 分析调用后端 /api/analyze，接收结构化 JSONF4能力画像可视化雷达图 + 任务列表 + 关键词 ChipF5被找回候选人高亮金色光效、动效呼吸标识 AI 复核推荐F6候选人详情解释卡证据 / 理由 / 风险 / 面试追问 4 TabF7风险与免责声明始终可见「AI 辅助复核，非唯一录用依据」F8失败兜底模型超时/格式错时切换缓存结果并明确标注F9离线 Demo 模式未配置 API Key 时自动加载缓存 JSONF10重置演示一键回到初始状态，便于录屏

四、交互流程（关键时序）

code

进入首页
  └─ 自动加载 sample JD + 候选人池（淡入动画）
点击「运行传统筛选」
  └─ 候选人卡片依次刷新状态徽章（带 stagger 动画）
  └─ 顶部统计「通过 X / 漏筛 Y」滚动数字
点击「启动潜力雷达」
  └─ 主按钮转为 Loading + 进度文案多阶段切换
  └─ AI 分析结果区淡入：先画雷达图 → 再出对照面板 → 再列出被找回候选人
  └─ 被找回候选人卡片在池中触发金色发光呼吸效果
点击任一被找回候选人
  └─ 右侧抽屉滑入，Tab 默认聚焦「能力证据」
  └─ 简历原文引用以高亮 mark 形式逐条出现
  └─ 可复制面试追问 / 加入面试池
异常路径
  └─ 模型超时 → Toast「调用失败，已切换缓存演示」+ 顶部状态徽章变橙
  └─ 无 API Key → 启动即进入离线 Demo 模式，徽章显示「Offline Demo」


五、设计语言（Design Language）

1. 整体风格

基调：科技感 + 招聘行业的专业感，避免冰冷或娱乐化

隐喻：雷达扫描、信号点亮、证据高亮（类似搜索 highlight）

质感：浅色为主、低饱和、微毛玻璃、柔和阴影；适度 glow 效果用于 AI 找回的候选人

2. 色彩系统（语义化 Token，全部写入 index.css 的 HSL 变量）

--primary：深空蓝（雷达扫描感）

--primary-glow：青色光晕（AI 高亮）

--accent：金色 / 琥珀（被找回候选人专属高亮）

--success：通过关键词的状态

--muted：被关键词漏筛的低优先态

--warning：风险提示

--destructive：错误兜底

梯度：

--gradient-radar：从 primary 到 primary-glow 的径向渐变（背景雷达感）

--gradient-card：候选人卡片微妙渐变

阴影：--shadow-elegant、--shadow-glow（用于 AI 找回卡片）

3. 字体

中英混排：英文 Inter / 中文系统无衬线

标题 600/700 字重，正文 400/500

数字与统计使用等宽数字（tabular-nums）增强专业感

4. 组件规范（基于 shadcn 变体扩展）

Button：新增 radar 变体（带 glow），outline 变体保证文字可见

Badge：hit / missed / rediscovered 三种状态变体

Card：evidence（带左侧色条引用）、spotlight（AI 找回高亮）

雷达图：使用 recharts RadarChart，统一描边色

Sheet（右侧抽屉）：宽度 560px，背景毛玻璃

5. 动效

状态切换：transition-smooth（0.3s cubic-bezier）

找回候选人：呼吸 glow（2s 循环）

Loading：进度文案 fade in/out 切换，避免静态等待

抽屉滑入：spring 缓动

数字滚动：使用 count-up

6. 响应式

桌面（≥1280px）：3 区并列（左 JD / 中候选人池 / 右 AI 结果），或上下分区

平板（768-1280）：候选人池 2 列，AI 结果在下

移动端：全部单列，操作按钮固定底部

7. 可达性

所有交互元素 hover/focus 状态明确

颜色对比度遵循 WCAG AA

状态不仅靠颜色区分，同时配合图标与文字标签

前端必须校验字段缺失情况并优雅降级；任何字段缺失时显示骨架占位而非崩溃。

七、明确的非目标（不要做）

不做登录、注册、多企业租户

不做简历文件上传/PDF 解析

不做多岗位管理与切换

不做真实 ATS 流转、Offer 发放、自动淘汰

不在前端存储任何 API Key

不在前端直接请求 Qwen，必须经由后端 /api/analyze
